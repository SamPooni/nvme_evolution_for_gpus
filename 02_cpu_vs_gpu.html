Key Insight: I/O Agent Warps

Not all 18,432 CUDA cores submit I/O commands. Only 1-64 I/O-agent warps actually interact with NVMe. The highlighted cores above represent these I/O agents — the rest are doing computation while waiting for data.

SIMT Execution Model

GPUs use Single Instruction, Multiple Threads (SIMT) — 32 threads form a warp and execute the same instruction in lockstep. When any thread in a warp stalls on I/O, the entire warp stalls.

The Stall Problem

When Warp 2 waits for NVMe I/O completion, all 32 threads are blocked — even if only one thread initiated the I/O. This is why interrupts are disastrous for GPUs and polling is preferred.

Why This Matters for NVMe

The Path Forward

NVMe protocol was designed assuming CPU-style interrupt handling. For GPUs, we need:

  • Polling-based completion — No interrupts, GPU checks completion queue
  • Large I/O coalescing — Batch small requests into large transfers
  • Direct GPU memory access — GPUDirect Storage bypasses CPU entirely
  • Queue depth optimization — Match NVMe queues to GPU warp count
// Traditional CPU I/O pattern (interrupt-driven) void cpu_read_file() { submit_nvme_command(cmd); while (!completion_interrupt) { sleep(); // CPU yields, waits for interrupt } process_data(); } // GPU I/O pattern (polling-based) __global__ void gpu_read_file() { submit_nvme_command(cmd); while (!check_completion_queue()) { // Busy-wait — entire warp spins here! __nanosleep(100); } process_data(); }

Thread Sync & Polling

How NVMe's interrupt-driven completion model clashes with GPU SIMT execution, and why polling is the path forward for GPU-initiated I/O.

Interrupt vs Polling: The Core Problem

CPUs are designed to efficiently handle interrupts — they can context switch in microseconds. GPUs have no such mechanism. When a GPU warp issues an I/O request, it has two choices: block or poll.

The SIMT Tax

When one thread in a GPU warp polls for I/O completion, all 32 threads in that warp are blocked. This is the fundamental cost of SIMT (Single Instruction, Multiple Threads) execution — lockstep means lockstep.

Efficient Polling Patterns

The key to efficient GPU polling is minimizing warp stalls while still detecting completions promptly. Here's the recommended pattern:

// Efficient GPU polling pattern for NVMe completions __device__ void poll_completion_efficient( volatile uint32_t* cq_head, uint16_t expected_phase ) { int poll_count = 0; const int POLL_BACKOFF_THRESHOLD = 100; while (true) { // Check completion queue entry uint32_t cq_entry = *cq_head; uint16_t phase = (cq_entry >> 16) & 0x1; if (phase == expected_phase) { // Completion received! return; } // Exponential backoff to reduce memory traffic if (++poll_count > POLL_BACKOFF_THRESHOLD) { __nanosleep(100); // Brief pause } // Memory fence to ensure fresh read __threadfence_system(); } } // Only I/O-agent warp polls; others continue computing __global__ void io_kernel() { if (threadIdx.x < 32) { // Warp 0 = I/O agent submit_nvme_command(); poll_completion_efficient(cq_ptr, phase); process_data(); } else { compute_work(); // Other warps keep working } }
Best Practice

Dedicate 1-2 warps per SM as I/O agents. These warps handle all storage operations while the remaining warps (typically 32-64 per SM) continue compute work. This maximizes GPU utilization during I/O.

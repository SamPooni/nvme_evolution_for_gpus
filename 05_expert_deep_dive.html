Key Insight: Software Overhead Dominates Small I/O

For 4KB reads, ~30% of latency is software overhead (syscall, driver, context switches). GDS eliminates most of this by bypassing the kernel. For large sequential I/O, NAND access dominates and the benefit is smaller.

NVMe Command Structure

Every NVMe I/O operation requires a 64-byte Submission Queue Entry (SQE) and produces a 16-byte Completion Queue Entry (CQE). Understanding these structures is key to optimizing queue management.

PCIe Transaction Flow

// GPUDirect Storage read - bypasses CPU entirely cudaError_t gds_read_example() { CUfileHandle_t fh; void* gpu_buffer; // Allocate GPU memory cudaMalloc(&gpu_buffer, BUFFER_SIZE); // Register buffer with GDS (one-time setup) cuFileBufRegister(gpu_buffer, BUFFER_SIZE, 0); // Open file with O_DIRECT for DMA int fd = open("/mnt/nvme/data.bin", O_RDONLY | O_DIRECT); cuFileHandleRegister(&fh, fd); // Direct read: NVMe → GPU (zero CPU copies!) ssize_t bytes = cuFileRead(fh, gpu_buffer, BUFFER_SIZE, 0, 0); // GPU can now process data immediately launch_kernel<<>>(gpu_buffer); return cudaSuccess; }

Bottom Line

GPUDirect Storage delivers 2× sequential throughput and 4.7× random IOPS while freeing the CPU for other work. The improvement is most dramatic for small I/O sizes where software overhead dominates.
